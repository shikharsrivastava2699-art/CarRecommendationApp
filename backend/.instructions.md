# Backend Code Review & Development Guidelines

## Overview
This document outlines the coding standards, best practices, and review guidelines for the CarRecommendationApp backend. All code changes should adhere to these guidelines.

---

## 1. API Design Standards

### 1.1 RESTful API Principles

#### Meaningful Resource-Based Routes
- **Use nouns, not verbs** - resources, not actions
- **Use HTTP methods correctly** - GET, POST, PUT, DELETE, PATCH
- **Include version in path** - `/api/v1/`
- **Use plural resource names** - `/cars`, not `/car`

✅ **Good:**
```typescript
// Reading resources
GET    /api/v1/cars              // List all cars
GET    /api/v1/cars/:id          // Get specific car
GET    /api/v1/cars/:id/reviews  // Get car's reviews

// Creating resources
POST   /api/v1/cars              // Create new car (admin only)
POST   /api/v1/reviews           // Submit review

// Updating resources
PUT    /api/v1/cars/:id          // Replace entire car
PATCH  /api/v1/cars/:id          // Update specific fields

// Deleting resources
DELETE /api/v1/cars/:id          // Delete car
DELETE /api/v1/reviews/:id       // Delete review

// Complex operations (RPC-style when necessary)
POST   /api/v1/recommendations   // Generate recommendations
POST   /api/v1/comparisons       // Compare multiple cars
```

❌ **Bad:**
```typescript
GET  /api/getCar/:id
POST /api/createCar
GET  /api/fetchReviews/:carId
POST /api/deleteReview/:reviewId
GET  /api/getRecommendationsForUser
```

#### Query Parameters
- **Use for filtering, sorting, pagination**
- **Consistent naming** across endpoints

✅ **Good:**
```typescript
GET /api/v1/cars?budget_min=1000000&budget_max=2500000
GET /api/v1/cars?vehicle_type=SUV&transmission=automatic
GET /api/v1/reviews?car_id=123&sort=rating&order=desc&page=1&limit=10
GET /api/v1/cars?search=camry&fields=make,model,price
```

---

### 1.2 HTTP Status Codes

Use correct status codes for different scenarios:

```typescript
// 2xx Success
200  - OK (GET, PUT, PATCH successful)
201  - Created (POST successful, resource created)
204  - No Content (DELETE successful, DELETE response)

// 4xx Client Error
400  - Bad Request (validation failed, invalid input)
401  - Unauthorized (authentication required)
403  - Forbidden (authenticated but not authorized)
404  - Not Found (resource doesn't exist)
409  - Conflict (resource conflict, e.g., duplicate)
422  - Unprocessable Entity (validation error details)

// 5xx Server Error
500  - Internal Server Error (unexpected error)
503  - Service Unavailable (maintenance, database down)
```

✅ **Good:**
```typescript
// Success responses
res.status(200).json({ data: cars });           // GET all
res.status(201).json({ data: newCar });         // POST create
res.status(204).send();                          // DELETE success

// Error responses
res.status(400).json({ error: 'Invalid budget range' });
res.status(401).json({ error: 'Authentication required' });
res.status(404).json({ error: 'Car not found' });
res.status(422).json({ errors: validationErrors });
```

#### Consistent JSON Structure
- **Always return consistent structure**
- **Include metadata for collections**
- **Include error details**

✅ **Good:**

## 2. TypeScript & Type Safety

### 2.1 Interfaces & Types for API

#### Request/Response Types
- **Define interfaces for all API contracts**
- **Separate DTOs (Data Transfer Objects) from domain models**
- **Use discriminated unions for complex responses**

✅ **Good:**
```typescript
// domain/car.ts

  id: string;
  make: string;
  model: string;
  variant: string;
  price: number;
  specs: CarSpecs;

  updatedAt: Date;

// dto/car.ts - API response format
export interface CarDTO {
  id: string;
  make: string;
  model: string;
  variant: string;
  price: number;
  specs: CarSpecsDTO;

}

// dto/api-response.ts

  status: 'success' | 'error';
  data?: T;
  message?: string;

}

export interface PaginatedResponse<T> {
  data: T[];
  meta: PaginationMeta;
}

// requests/quiz.ts
```

### 2.2 Type Safety Best Practices
- **Never use `any`** - use `unknown` and narrow types
- **Use `readonly` for immutable data**
- **Use enums or literal types** for fixed sets of values

```typescript
// ✅ Good - Enum for status
export enum ReviewStatus {
  PENDING = 'pending',
  APPROVED = 'approved',
  REJECTED = 'rejected'
}

// ✅ Good - Union types
export type FuelType = 'petrol' | 'diesel' | 'hybrid' | 'electric';
export type Transmission = 'manual' | 'automatic' | 'cvt';

// ✅ Good - Discriminated union for responses
export type QuizResponse = 
  | { status: 'success'; recommendations: Car[] }
  | { status: 'error'; message: string };

// ❌ Bad - Using any
function processReview(review: any) {}

// ❌ Bad - String literals repeated
if (car.fuelType === 'petrol' || car.fuelType === 'diesel') {}
```

## 4. Service & Controller Architecture

### 4.1 Separation of Concerns

**Controller** - Handles HTTP request/response
```typescript
// controllers/car.controller.ts
export class CarController {
  constructor(private carService: CarService) {}

  async getAllCars(req: Request, res: Response): Promise<void> {
    try {
      const filters = req.query;
      const cars = await this.carService.searchCars(filters);
      
      res.status(200).json({
        status: 'success',
        data: cars
      });
    } catch (error) {
      res.status(500).json({
        status: 'error',
        message: 'Failed to fetch cars'
      });
    }
  }

  async getCarById(req: Request, res: Response): Promise<void> {
    try {
      const { id } = req.params;
      const car = await this.carService.getCar(id);

      if (!car) {
        res.status(404).json({
          status: 'error',
          message: 'Car not found',
          code: 'CAR_NOT_FOUND'
        });
        return;
      }

      res.status(200).json({
        status: 'success',
        data: car
      });
    } catch (error) {
      res.status(500).json({
        status: 'error',
        message: 'Failed to fetch car'
      });
    }
  }
}
```

**Service** - Contains business logic
```typescript
// services/car.service.ts
export class CarService {
  constructor(private carRepository: CarRepository) {}

  async getCar(id: string): Promise<Car | null> {
    // Business logic here
    const car = await this.carRepository.findById(id);
    return car;
  }

  async searchCars(filters: SearchFilters): Promise<Car[]> {
    // Apply filters, sorting, pagination
    const cars = await this.carRepository.search(filters);
    return cars;
  }

  async compareCars(carIds: string[]): Promise<ComparisonResult> {
    // Business logic for comparison
    const cars = await this.carRepository.findByIds(carIds);
    return this.formatComparison(cars);
  }

  private formatComparison(cars: Car[]): ComparisonResult {
    // Format comparison result
    return { cars, categories: [] };
  }
}
```

**Repository** - Data access layer
```typescript
// repositories/car.repository.ts
export class CarRepository {
  constructor(private db: Database) {}

  async findById(id: string): Promise<Car | null> {
    return this.db.cars.findOne({ id });
  }

  async findByIds(ids: string[]): Promise<Car[]> {
    return this.db.cars.find({ id: { $in: ids } });
  }

  async search(filters: SearchFilters): Promise<Car[]> {
    let query = this.db.cars;
    
    if (filters.budgetMin) {
      query = query.where('price').gte(filters.budgetMin);
    }
    if (filters.budgetMax) {
      query = query.where('price').lte(filters.budgetMax);
    }
    if (filters.vehicleType) {
      query = query.where('bodyType').in(filters.vehicleType);
    }
    
    return query.exec();
  }
}
```

### 4.2 Error Handling

**Custom Error Classes**
```typescript
// utils/errors.ts
export class AppError extends Error {
  constructor(
    public message: string,
    public statusCode: number,
    public code: string
  ) {
    super(message);
    this.name = 'AppError';
  }
}

export class ValidationError extends AppError {
  constructor(message: string, public errors?: Record<string, string>) {
    super(message, 422, 'VALIDATION_ERROR');
  }
}

export class NotFoundError extends AppError {
  constructor(resource: string) {
    super(`${resource} not found`, 404, `${resource.toUpperCase()}_NOT_FOUND`);
  }
}

export class UnauthorizedError extends AppError {
  constructor(message = 'Authentication required') {
    super(message, 401, 'UNAUTHORIZED');
  }
}

export class ForbiddenError extends AppError {
  constructor(message = 'Access denied') {
    super(message, 403, 'FORBIDDEN');
  }
}
```

**Error Handling Middleware**
```typescript
// middleware/error-handler.middleware.ts
export function errorHandler(
  err: Error,
  req: Request,
  res: Response,
  next: NextFunction
): void {
  console.error('Error:', err);

  if (err instanceof AppError) {
    res.status(err.statusCode).json({
      status: 'error',
      message: err.message,
      code: err.code,
      ...(err instanceof ValidationError && { errors: err.errors })
    });
    return;
  }

  // Unknown error
  res.status(500).json({
    status: 'error',
    message: 'Internal server error',
    code: 'INTERNAL_ERROR'
  });
}
```

## 5. Input Validation

### 5.1 Validation Patterns

**Middleware-based validation**
```typescript
// middleware/validation.middleware.ts
import { body, query, param, validationResult } from 'express-validator';

export const validateQuizRequest = [
  body('budgetMin').isInt({ min: 0 }).withMessage('Budget min must be positive'),
  body('budgetMax').isInt({ min: 0 }).withMessage('Budget max must be positive'),
  body('vehicleTypes').isArray({ min: 1 }).withMessage('Select at least one vehicle type'),
  body('primaryUseCase').isString().notEmpty(),
  body('priorities').isArray({ min: 1, max: 3 }).withMessage('Select 1-3 priorities'),
  (req: Request, res: Response, next: NextFunction) => {
    const errors = validationResult(req);
    if (!errors.isEmpty()) {
      throw new ValidationError('Invalid request', errors.mapped());
    }
    next();
  }
];

export const validateGetCar = [
  param('id').isString().notEmpty().withMessage('Car ID is required'),
  (req: Request, res: Response, next: NextFunction) => {
    const errors = validationResult(req);
    if (!errors.isEmpty()) {
      throw new ValidationError('Invalid request', errors.mapped());
    }
    next();
  }
];

export const validateSearchQuery = [
  query('budgetMin').optional().isInt({ min: 0 }),
  query('budgetMax').optional().isInt({ min: 0 }),
  query('page').optional().isInt({ min: 1 }),
  query('limit').optional().isInt({ min: 1, max: 100 }),
  (req: Request, res: Response, next: NextFunction) => {
    const errors = validationResult(req);
    if (!errors.isEmpty()) {
      throw new ValidationError('Invalid query parameters', errors.mapped());
    }
    next();
  }
];
```

## 6. Async/Await & Promise Handling

### 6.1 Best Practices
- **Always use async/await** - more readable than .then()
- **Wrap async operations in try/catch**
- **Use Promise.all() for parallel operations**
- **Avoid nested async operations**

✅ **Good:**
```typescript
// Parallel operations
async function getUserWithCars(userId: string): Promise<{ user: User; cars: Car[] }> {
  const [user, cars] = await Promise.all([
    userService.getUser(userId),
    carService.getUserCars(userId)
  ]);
  return { user, cars };
}

// Sequential with error handling
async function processRecommendationQuiz(quizData: QuizRequest): Promise<Car[]> {
  try {
    // Validate input
    validateQuizData(quizData);

    // Fetch matching cars
    const cars = await this.carService.searchCars(quizData);

    // Score and rank
    const scored = cars.map(car => ({
      ...car,
      matchScore: this.calculateScore(car, quizData)
    }));

    return scored.sort((a, b) => b.matchScore - a.matchScore).slice(0, 10);
  } catch (error) {
    throw new AppError('Failed to process quiz', 500, 'QUIZ_PROCESSING_ERROR');
  }
}
```

❌ **Bad:**
```typescript
// Nested promises
function processRecommendationQuiz(quizData: QuizRequest) {
  return carService.searchCars(quizData)
    .then(cars => {
      return Promise.all(cars.map(car => {
        return reviewService.getReviews(car.id)
          .then(reviews => ({ ...car, reviews }));
      }));
    })
    .then(carsWithReviews => {
      return carsWithReviews.map(car => ({
        ...car,
        matchScore: calculateScore(car, quizData)
      }));
    });
}

// Sequential without parallelization
async function processRecommendationQuiz(quizData: QuizRequest) {
  const cars = await carService.searchCars(quizData);
  for (const car of cars) {
    const reviews = await reviewService.getReviews(car.id);  // Slow!
    // process...
  }
}
```

## 7. Database Best Practices

### 7.1 Query Optimization
- **Use indexes** for frequently queried fields
- **Limit fields** returned from queries
- **Use pagination** for large datasets
- **Avoid N+1 queries** - use joins/populate

✅ **Good:**
```typescript
// Select only needed fields
const cars = await this.db.cars
  .find({ vehicleType: 'SUV' })
  .select('id make model price')
  .limit(10);

// Use indexes
// In migration: CREATE INDEX idx_budget ON cars(price);
const carsInBudget = await this.db.cars
  .find({ price: { $gte: min, $lte: max } });

// Pagination
const page = req.query.page || 1;
const limit = 10;
const skip = (page - 1) * limit;

const [data, total] = await Promise.all([
  this.db.cars.find().skip(skip).limit(limit),
  this.db.cars.countDocuments()
]);

// Use populate to avoid N+1
const cars = await this.db.cars.find().populate('reviews');
```

❌ **Bad:**
```typescript
// Fetching all fields
const cars = await this.db.cars.find();

// No pagination on large queries
const allCars = await this.db.cars.find({ vehicleType: 'SUV' });

// N+1 problem
const cars = await this.db.cars.find();
for (const car of cars) {
  car.reviews = await this.db.reviews.find({ carId: car.id });  // N queries!
}
```

## 10. Common Pitfalls to Avoid

❌ **Don't:**
- Mix business logic with route handlers
- Use any for types
- Forget error handling
- Use string literals for repeated values (use enums)
- Make infinite loops in queries
- Expose sensitive data in error messages
- Use synchronous operations
- Hardcode configuration values
- Skip input validation
- Leave console.log statements

✅ **Do:**
- Keep controllers thin, services thick
- Use explicit types everywhere
- Handle all error cases
- Use constants/enums for fixed values
- Use pagination and limits
- Log errors, not sensitive data
- Use async/await consistently
- Use environment variables for config
- Validate all user input
- Use proper logging framework

## 11. Logging Standards

### 11.1 Logger Setup

```typescript
// utils/logger.ts
export enum LogLevel {
  DEBUG = 'debug',
  INFO = 'info',
  WARN = 'warn',
  ERROR = 'error'
}

export class Logger {
  private static instance: Logger;

  static getInstance(): Logger {
    if (!this.instance) {
      this.instance = new Logger();
    }
    return this.instance;
  }

  debug(message: string, meta?: Record<string, any>): void {
    console.log(`[DEBUG] ${message}`, meta);
  }

  info(message: string, meta?: Record<string, any>): void {
    console.log(`[INFO] ${message}`, meta);
  }

  warn(message: string, meta?: Record<string, any>): void {
    console.warn(`[WARN] ${message}`, meta);
  }

  error(message: string, error?: Error, meta?: Record<string, any>): void {
    console.error(`[ERROR] ${message}`, error?.message, meta);
  }
}

export const logger = Logger.getInstance();
```

### 11.1 Usage

```typescript
// In service
logger.info('Fetching cars', { filters });
logger.debug('Calculated match score', { carId, score });
logger.warn('Database slow query', { queryTime: 500 });
```

```typescript
      const car: Car = { id: '1', make: 'Toyota', ... };
      mockRepository.findById.mockResolvedValue(car);

      const result = await carService.getCar('1');

      expect(result).toEqual(car);
      expect(mockRepository.findById).toHaveBeenCalledWith('1');
    });

    it('should return null if car not found', async () => {
```

## Summary

**Key Principles:**
1. **RESTful Design** - Use resources, proper HTTP methods
2. **Type Safety** - Always use explicit types, avoid `any`
3. **Separation of Concerns** - Thin controllers, thick services
4. **Error Handling** - Use custom errors, proper status codes
5. **Input Validation** - Validate all user input
6. **Performance** - Optimize queries, use pagination
7. **Documentation** - JSDoc for public methods
8. **Testing** - Test business logic thoroughly
9. **Security** - Protect sensitive data, sanitize input
10. **Logging** - Log appropriately, not sensitive data

**Before every PR, ask:**
- Are API endpoints RESTful?
- Is all input validated?
- Are errors handled properly?
- Are database queries optimized?
- Is sensitive data protected?
- Is code testable and tested?

If the answer to any is "no", refactor before submitting.
