# Frontend Code Review & Development Guidelines

## Overview
This document outlines the coding standards, best practices, and review guidelines for the CarRecommendationApp frontend. All code changes should adhere to these guidelines.

---

## 1. Code Quality Standards

### 1.1 Naming Conventions

#### Variables & Functions
- **Use camelCase** for variables and function names
- Start with descriptive verbs for functions (`get`, `fetch`, `handle`, `calculate`, etc.)
- Avoid single-letter variables except in loops

✅ **Good:**
```typescript
const userId = getUserId();
const isLoading = true;
function handleInputChange(event: React.ChangeEvent<HTMLInputElement>) {}
function fetchCarRecommendations(quizData: QuizData) {}
```

❌ **Bad:**
```typescript
const u = getUserId();
const load = true;
const x = event.target.value;
function f(e) {}
```

#### Constants
- **Use UPPER_SNAKE_CASE** for constants
- Group related constants together

✅ **Good:**
```typescript
const API_BASE_URL = 'https://api.example.com';
const MAX_RECOMMENDATIONS = 10;
const QUIZ_STEPS = 6;
```

#### Component Names
- **Use PascalCase** for React components
- Use descriptive names that indicate component purpose

✅ **Good:**
```typescript
function CarCard() {}
function RecommendationQuiz() {}
function ComparisonTable() {}
```

❌ **Bad:**
```typescript
function carCard() {}
function Quiz() {}
function Table() {}
```

#### File Names
- **Use kebab-case** for file names
- Match file name to primary export

✅ **Good:**
```
car-card.tsx
recommendation-quiz.tsx
api-service.ts
```

❌ **Bad:**
```
CarCard.tsx
recommendationQuiz.tsx
service.ts
```

### 1.2 TypeScript Best Practices

#### Interfaces & Types
- **Always use explicit types** - avoid `any` at all costs
- Create dedicated type/interface files for domain models
- Use `interface` for object shapes, `type` for unions and primitives

✅ **Good:**
```typescript
// types/car.ts
export interface Car {
  id: string;
  make: string;
  model: string;
  price: number;
  specs: CarSpecs;
}

export interface CarSpecs {
  engine: string;
  transmission: 'manual' | 'automatic';
  fuelType: 'petrol' | 'diesel' | 'hybrid';
}

// components/car-card.tsx
interface CarCardProps {
  car: Car;
  onCompare: (carId: string) => void;
}

function CarCard({ car, onCompare }: CarCardProps) {
  return <div>...</div>;
}
```

❌ **Bad:**
```typescript
// Inline, vague types
function CarCard({ car, onCompare }: any) {}

// Missing type information
const cars = [];  // What's in this array?

// Using 'any' type
function handleResponse(data: any) {}
```

#### Type Safety
- **Avoid `any` type** - use `unknown` if truly unknown, then narrow type
- **Use `readonly` for immutable data**
- **Use discriminated unions** for complex type logic

✅ **Good:**
```typescript
export interface SuccessResponse<T> {
  status: 'success';
  data: T;
}

export interface ErrorResponse {
  status: 'error';
  message: string;
}

type ApiResponse<T> = SuccessResponse<T> | ErrorResponse;

function handleResponse<T>(response: ApiResponse<T>) {
  if (response.status === 'success') {
    console.log(response.data);
  } else {
    console.error(response.message);
  }
}
```

### 1.3 React Best Practices

#### Functional Components
- **Always use functional components** (never class components)
- **Use React Hooks** for state and lifecycle management
- **Extract custom hooks** for reusable logic

✅ **Good:**
```typescript
function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(false);

  useEffect(() => {
    const fetchUser = async () => {
      setLoading(true);
      const data = await userService.getUser(userId);
      setUser(data);
      setLoading(false);
    };

    fetchUser();
  }, [userId]);

  if (loading) return <LoadingSpinner />;
  if (!user) return <div>User not found</div>;

  return <div>{user.name}</div>;
}
```

#### Props Pattern
- **Destructure props** in function parameters
- **Create separate interface for props**
- **Avoid prop drilling** - use Context API or state management for deeply nested props

✅ **Good:**
```typescript
interface SubmitButtonProps {
  label: string;
  onClick: () => void;
  isLoading?: boolean;
  disabled?: boolean;
}

function SubmitButton({
  label,
  onClick,
  isLoading = false,
  disabled = false
}: SubmitButtonProps) {
  return (
    <button onClick={onClick} disabled={disabled || isLoading}>
      {isLoading ? 'Loading...' : label}
    </button>
  );
}
```

#### State Management
- **Keep state as local as possible**
- **Use React Context** for app-wide state (auth, theme)
- **Lift state only when necessary**
- **Avoid unnecessary re-renders** using `useMemo` and `useCallback`

### 1.5 Unused Code Removal

#### Variables
- **Remove all unused variables** before committing
- **Use TypeScript strict mode** to catch unused variables
- **Review all `const` declarations**

✅ **Good:**
```typescript
const cars = await carService.getCars();
const topCars = cars.slice(0, 10);
return topCars;
```

❌ **Bad:**
```typescript
const cars = await carService.getCars();
const allCars = cars;  // unused
const topCars = cars.slice(0, 10);
const temp = topCars;  // unused
return topCars;
```

#### Imports
- **Remove unused imports** - use TypeScript `noUnusedLocals`
- **Remove commented-out code** - use version control instead

✅ **Good:**
```typescript
import { useEffect, useState } from 'react';
import { carService } from '../services/car-service';
import { Car } from '../types/car';
```

❌ **Bad:**
```typescript
import { useEffect, useState, useCallback } from 'react';  // useCallback unused
import { carService } from '../services/car-service';
import { Car } from '../types/car';
import { User } from '../types/user';  // unused
// const unused = false;
```

#### Dead Code
- **Remove unreachable code**
- **Remove debug console.log statements** (except in dev mode)
- **Remove old/abandoned functions**

## 2. Code Review Checklist

### Before Submitting PR:

- [ ] **No TypeScript errors** - `npm run type-check` passes
- [ ] **No ESLint errors** - `npm run lint` passes
- [ ] **No unused variables or imports**
- [ ] **All props properly typed** with interfaces
- [ ] **Component props follow pattern** (destructured, interface defined)
- [ ] **No hardcoded values** - use constants
- [ ] **Error handling implemented** - try/catch or error boundaries
- [ ] **Loading states handled** - show spinners/skeletons
- [ ] **Responsive design tested** - works on mobile, tablet, desktop
- [ ] **Accessibility checked** - proper labels, ARIA attributes
- [ ] **No console.log statements** left in code
- [ ] **Code is DRY** - no duplication, reuse components
- [ ] **Functions have single responsibility**
- [ ] **Meaningful variable names** - searchable, descriptive
- [ ] **Comments only for why, not what** - code should be self-documenting

## 3. Performance Guidelines

### 3.1 Rendering Optimization
- **Memoize expensive components** using `React.memo`
- **Split code** using route-based code splitting
- **Lazy load images** and components
- **Avoid inline functions** in render - use `useCallback`

```typescript
interface ListItemProps {
  item: Item;
  onSelect: (id: string) => void;
}

// Memoized to prevent unnecessary re-renders
const ListItem = React.memo(function ListItem({
  item,
  onSelect
}: ListItemProps) {
  const handleClick = useCallback(() => {
    onSelect(item.id);
  }, [item.id, onSelect]);

  return <button onClick={handleClick}>{item.name}</button>;
});
```

### 3.2 Bundle Size
- **Tree-shake unused exports**
- **Use dynamic imports** for heavy components
- **Avoid importing entire libraries** when only needing parts

```typescript
// ❌ Bad - imports entire lodash library
import _ from 'lodash';

// ✅ Good - imports only what's needed
import debounce from 'lodash-es/debounce';
```

## 4. Testing Guidelines

### 4.1 Unit Tests
- **Test component logic** - props, state, event handlers
- **Mock external services** - API calls, localStorage
- **Aim for 80%+ coverage** of critical paths

```typescript
import { render, screen, fireEvent } from '@testing-library/react';
import { CarCard } from './car-card';

describe('CarCard', () => {
  it('should display car details', () => {
    const car: Car = { id: '1', make: 'Toyota', model: 'Camry', price: 2500000 };
    render(<CarCard car={car} onCompare={() => {}} />);

    expect(screen.getByText('Toyota Camry')).toBeInTheDocument();
    expect(screen.getByText('₹2500000')).toBeInTheDocument();
  });

  it('should call onCompare when compare button clicked', () => {
    const car: Car = { id: '1', make: 'Toyota', model: 'Camry', price: 2500000 };
    const onCompare = jest.fn();
    render(<CarCard car={car} onCompare={onCompare} />);

    fireEvent.click(screen.getByText('Compare'));
    expect(onCompare).toHaveBeenCalledWith('1');
  });
});
```

## 6. Error Handling

### 6.1 Try-Catch Pattern
- **Always handle API errors**
- **Show meaningful error messages to users**
- **Log errors for debugging**

```typescript
function useCarData(carId: string) {
  const [car, setCar] = useState<Car | null>(null);
  const [error, setError] = useState<string | null>(null);
  const [loading, setLoading] = useState(false);

  useEffect(() => {
    const fetchCar = async () => {
      try {
        setLoading(true);
        setError(null);
        const data = await carService.getCar(carId);
        setCar(data);
      } catch (err) {
        const message = err instanceof Error ? err.message : 'Failed to load car';
        setError(message);
        console.error('Error fetching car:', err);
      } finally {
        setLoading(false);
      }
    };

    fetchCar();
  }, [carId]);

  return { car, error, loading };
}
```

## 7. Git Commit Messages

### Format
```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type
- `feat`: New feature
- `fix`: Bug fix
- `refactor`: Code restructuring without behavior change
- `style`: Formatting, missing semicolons, etc.
- `test`: Adding/updating tests
- `docs`: Documentation updates
- `chore`: Maintenance, dependencies

### Example
```
feat(quiz): add feature priority ranking step

- Add drag-and-drop ranking UI component
- Implement priority weighting in recommendation algorithm
- Update quiz data structure to include ranked priorities
- Add validation for 1-3 priority selection

Fixes #42
```

## 8. Development Workflow

### Before Starting
1. Create a new branch: `git checkout -b feat/feature-name`
2. Sync with main: `git pull origin main`

### During Development
1. Write code following this guide
2. Run tests: `npm run test`
3. Check types: `npm run type-check`
4. Run linter: `npm run lint -- --fix`

### Before Committing
1. Review your changes: `git diff`
2. Stage changes: `git add .`
3. Run full checks:
   ```bash
   npm run type-check
   npm run lint
   npm run test
   ```
4. Commit with descriptive message

### Creating PR
1. Push branch: `git push origin feat/feature-name`
2. Create PR with:
   - Clear description of changes
   - Link to related issues
   - Screenshots for UI changes
   - Checklist completion

## 9. Common Pitfalls to Avoid

❌ **Don't:**
- Use `any` type
- Create prop drilling chains (>3 levels deep)
- Mutate state directly: `state.property = value`
- Use `index` as React key in lists
- Call hooks conditionally
- Forget to add dependencies to `useEffect`
- Mix styled-components with CSS Modules

✅ **Do:**
- Use explicit types
- Use Context API for shared state
- Create new objects/arrays for state updates
- Use unique, stable IDs as keys
- Always call hooks at top level
- Include all dependencies in dependency array
- Pick one styling approach and stick with it

## Summary

**Key Principles:**
1. **Type Safety First** - Always use proper types, avoid `any`
2. **Modularity** - Small, focused, reusable components
3. **Clarity** - Self-documenting code with clear naming
4. **Performance** - Optimize renders, bundle size
5. **Testing** - Write testable code, aim for good coverage
6. **Accessibility** - Make apps usable for everyone
7. **Consistency** - Follow patterns established in codebase

**Before every PR, ask:**
- Is this code readable?
- Is this code maintainable?
- Is this code tested?
- Is this code performant?
- Does this follow our guidelines?

If the answer to any is "no", refactor before submitting.
